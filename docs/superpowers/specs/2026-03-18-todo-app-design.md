# TODOアプリ設計書

## 概要

React の学習用として、最小構成の TODO アプリを作る。対象機能は TODO の追加、完了切替、削除に限定し、ブラウザの `localStorage` に永続化する。PC とスマホの両方で自然に使える 1 ページ構成とし、見た目は柔らかいトーンの `Calm Panel` をベースにする。

## 目的

- `useState` による状態管理の基本を理解できること
- フォーム入力と送信処理の流れを追えること
- 配列の追加、更新、削除を React の状態更新として読めること
- `useEffect` を使った `localStorage` 連携の基本を理解できること
- UI と状態管理の責務を分けた小さな構成を学べること

## スコープ

### 含めるもの

- TODO の新規追加
- TODO の完了状態の切替
- TODO の削除
- `localStorage` への保存と復元
- PC / スマホの両対応
- 学習しやすさを優先した軽いコンポーネント分割

### 含めないもの

- 編集機能
- 絞り込み機能
- 期限、優先度、カテゴリ
- サーバ保存
- ユーザー認証

## 画面設計

アプリは 1 ページのみとし、中央に 1 枚のカードを置く。カードの中に入力欄、追加ボタン、TODO 一覧を上から下へ並べ、視線移動が少ない構成にする。背景や配色は落ち着いた暖色寄りにして、学習用サンプルとして過度に装飾せず、読みやすさを優先する。

レスポンシブ対応は最小限でよい。PC では中央寄せの固定幅に近い見え方、スマホではカード幅が画面に合わせて自然に縮む見え方とする。

## 構成

責務は次の単位に分ける。

- `App`
  - ページ全体のレイアウトを持つ
  - 学習対象の入口として、画面の骨格だけを担当する
- `TodoForm`
  - 入力欄と追加操作を担当する
  - ユーザー入力の送信を受け取り、親またはフックにイベントを渡す
- `TodoList`
  - TODO 一覧を表示する
  - 空状態の表示有無もここで扱う
- `TodoItem`
  - 1 件の TODO 表示、完了切替、削除を担当する
- `useTodos`
  - TODO 配列の状態管理を担当する
  - 追加、完了切替、削除の操作関数を提供する
  - `localStorage` の読み込みと保存を担当する

この構成により、UI を読むときはコンポーネントを、状態の流れを読むときは `useTodos` を見ればよい状態にする。

## データ設計

TODO データは最小限の 3 項目に絞る。

```ts
type Todo = {
  id: string;
  title: string;
  completed: boolean;
};
```

- `id`
  - TODO を一意に識別するために使う
- `title`
  - 入力されたタスク名を保持する
- `completed`
  - 完了状態を表す

## 状態と操作

`useTodos` は TODO 配列を単一の状態として保持する。提供する操作は次の 3 つに限定する。

- `addTodo(title)`
  - 入力値を `trim` したうえで空文字を弾く
  - 新しい TODO を作り、配列に追加する
- `toggleTodo(id)`
  - 対象 TODO の `completed` を反転する
- `removeTodo(id)`
  - 対象 TODO を配列から削除する

これにより、学習時に配列の追加、更新、削除がそれぞれ独立して確認できる。

## 永続化

初回表示時に `localStorage` から保存済みデータを読み込み、状態更新のたびに同じキーへ保存する。保存キーは固定文字列を 1 つ使う。読み込みに失敗した場合は空配列で開始し、保存に失敗した場合も UI の利用は継続できるようにする。

`localStorage` は副作用なので、読み込みと保存の責務は `useTodos` に閉じ込める。UI コンポーネント側では保存手段を意識しない。

## エラーハンドリング

学習用サンプルとして、エラー処理は過剰にしない。

- 空文字は追加しない
- `localStorage` の読み込み失敗時は空配列で継続する
- `localStorage` の保存失敗時はアプリ全体を止めない
- 必要に応じて `console.error` を使い、副作用の失敗箇所を把握しやすくする

## テスト方針

最低限、次の振る舞いを確認できるようにする。

- 初期表示で入力欄と空状態、または保存済み TODO が表示される
- TODO を追加できる
- TODO の完了状態を切り替えられる
- TODO を削除できる
- 保存済み TODO が次回表示時に復元される

テスト手段は UI 観点のコンポーネントテストを想定し、実装計画で採用するテスト基盤に合わせて具体化する。

## 受け入れ条件

- ユーザーが TODO を追加、完了、削除できる
- リロード後も `localStorage` から TODO が復元される
- PC / スマホの両方で 1 画面として無理なく閲覧できる
- UI と状態管理の責務が分かれていて、学習対象が追いやすい
- 今回スコープ外の機能が混ざっていない
